# Stand E2E Testing — проект и референс

Каталог содержит проект системы «стенды живых проектов как часть e2e-тестирования»
и референс-реализацию прототипа, на котором она основана.

## Файлы

- **[stand-e2e-plan.md](stand-e2e-plan.md)** — детальный проект архитектуры: multi-VM
  топология (stand+postgres / PACS / worker / host), фикстур-пайплайн (capture → anonymize →
  package → registry → load), обобщённый харнесс, интеграция в `test-all-stages`, фазы реализации,
  зафиксированные решения и риски.
- **[prototype-nir_liver/](prototype-nir_liver/)** — рабочий прототип workflow-теста для проекта
  `nir_liver`, на котором основан обобщённый харнесс. **Референс-снимок, не активный CI-тест**
  (захардкожен под `nir_liver`; секреты и целевые хосты — только через env-переменные):
  - `test_workflow.py` — сценарий-DAG (13 стадий хребта, от поступления исследования до патоморфологии)
  - `workflow_stand.py` — host-side driver `Stand` (httpx + ssh/scp + ожидания состояний/данных)
  - `stand_tool.py` — VM-side tool (резолв путей через `FileRepository`, геометрия NIfTI, синтез `.seg.nrrd`)
  - `setup_stand.sh` — провижн стенда (c-get, dicom-worker, регистрация Orthanc-модальности)
  - `conftest.py`, `README.md` — фикстуры и описание прототипа

## Статус реализации

| Компонент | План | Где реализуется |
|---|---|---|
| **Анонимизатор БД** (`clarinet anon scrub-db`) | §3.2 | с машины, подключённой к **продакшену** (нужен доступ к живой БД) |
| **Скраппер фикстур** (capture: БД + DICOM через DICOMweb-прокси + реальные пользовательские сегментации) | §3.3 | там же — с **прод-машины** (DICOMweb-прокси + `FileRepository`) |
| Multi-VM провижн / wiring | §3.1, фаза 1 | в репозитории |
| Обобщённый харнесс + формат сценария | §3.4–3.5, фаза 4 | в репозитории |
| CI-интеграция (`make test-stands`, nightly) | §4, фаза 6 | в репозитории |

Зафиксированные решения (план §6): хранилище фикстур — checksum в репо + артефакты на локальном
диске → позже S3; место в CI — отдельный nightly `make test-stands` (вне 40-минутного `test-all-stages`).

> Воркер на отдельной VM ходит в данные только по **HTTP (API + service-token)** и **RabbitMQ**, а не
> в PostgreSQL напрямую; файловые артефакты — через **общий storage (NFS)**. Детали — план §3.1.
